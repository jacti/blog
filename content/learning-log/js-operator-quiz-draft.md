# 퀴즈 구조

각 문항: **코드 제시 → 객관식/단답형 → 정답 공개 → 설명 + 실행 가능 에디터**

---

## Q1. `+true`의 결과는?

**보기**: `1` / `NaN` / `"true"` / `TypeError`
**정답**: `1`

### 설명

`+`가 숫자 앞에 붙는 건 익숙하죠? `+3`처럼요. 그런데 `+true`는 뭘까요?

`+` 단항 연산자는 사실 **숫자 변환기**예요. 피연산자를 숫자 타입으로 바꿔주는 게 유일한 역할이에요. 내부적으로 `Number()` 함수와 완전히 같은 동작을 해요 — 둘 다 ECMAScript 스펙의 `ToNumber()`라는 추상 연산을 호출하거든요.

그래서 `+true`는 `Number(true)`와 동일하고, 스펙에서 `true`는 `1`, `false`는 `0`으로 정의하고 있어요.

```js
+true       // 1
+false      // 0
+null       // 0
+undefined  // NaN — null은 0인데 undefined는 NaN이에요. 이건 그냥 외워야 해요.
+'42'       // 42
+'42px'     // NaN — 숫자가 아닌 문자가 섞이면 NaN
```

그러면 `Number()`가 있는데 왜 `+`가 따로 있을까요? 기능은 같지만 **짧게 쓸 수 있다**는 장점이 있어요. 실무에서 이런 패턴을 종종 보게 돼요:

```js
const timestamp = +new Date();  // 날짜 → 밀리초 타임스탬프
const num = +inputStr;          // 문자열 → 숫자
```

## Q2. `+[]`의 결과는?

**보기**: `0` / `NaN` / `""` / `TypeError`
**정답**: `0`

### 설명

빈 배열에 `+`를 붙이면 `0`이 나와요. 직관적이지 않죠? 이걸 이해하려면 JavaScript가 객체를 원시값으로 바꾸는 과정을 알아야 해요.

`+` 단항 연산자가 배열을 만나면 이런 순서로 변환이 일어나요:

1. `ToPrimitive([])` 호출 → 배열의 `toString()` 실행
2. `[].toString()` → `""` (빈 문자열)
3. `ToNumber("")` → `0`

빈 문자열이 `0`이 되는 건 스펙에 정의된 규칙이에요. 그래서 최종적으로 `0`이 나오는 거예요.

배열의 요소 개수에 따라 결과가 달라지는 것도 재미있어요:

```js
+[]      // 0    — [].toString() → "" → 0
+[1]     // 1    — [1].toString() → "1" → 1
+[1, 2]  // NaN  — [1,2].toString() → "1,2" → 숫자 파싱 불가 → NaN
```

핵심은 배열이 직접 숫자가 되는 게 아니라, **toString()을 거쳐 문자열이 된 다음 숫자로 변환**된다는 거예요. 중간에 문자열을 거친다는 걸 알면 결과를 예측할 수 있어요.

## Q3. `false + '1'`의 결과는?

**보기**: `"false1"` / `1` / `"01"` / `NaN`
**정답**: `"false1"`

### 설명

이항 `+` 연산자(양쪽에 피연산자가 있는 `+`)는 두 가지 완전히 다른 일을 해요: **숫자 덧셈** 또는 **문자열 연결**. 어느 쪽으로 동작할지는 이 규칙으로 결정돼요:

1. 양쪽 피연산자에 `ToPrimitive()` 적용
2. **둘 중 하나라도 문자열이면** → 나머지도 `ToString()`으로 변환 → **문자열 연결**
3. **둘 다 문자열이 아니면** → 둘 다 `ToNumber()` → **숫자 덧셈**

`false + '1'`에서 `'1'`이 문자열이니까 문자열 모드가 돼요. 그러면 `false`도 `ToString()`이 적용돼서 `"false"`가 되고, 최종적으로 `"false" + "1"` → `"false1"`이 되는 거예요.

`"01"`이 아니냐고요? `false`가 먼저 `0`으로 변환된 다음 문자열이 될 거라고 생각하기 쉽지만, 스펙의 순서는 **문자열 여부를 먼저 확인**해요. 문자열이 발견되면 `ToNumber`가 아니라 `ToString`이 적용되기 때문에 `false` → `"false"`가 돼요.

```js
false + '1'      // "false1"  — false → "false"
true + '1'       // "true1"   — true → "true"
null + '1'       // "null1"   — null → "null"
undefined + '1'  // "undefined1"

// 문자열이 없으면? 숫자 덧셈
false + 1        // 1  — false → 0, 숫자끼리 덧셈
null + 1         // 1  — null → 0
```

## Q4. `1 + 2 + '3'`의 결과는?

**보기**: `"123"` / `"33"` / `6` / `"6"`
**정답**: `"33"`

### 설명

`"123"`이라고 답하기 쉬운 함정 문제예요. 핵심은 `+` 연산자가 **왼쪽에서 오른쪽으로** 결합한다는 거예요 (좌결합성).

한 단계씩 따라가 볼게요:

```
1 + 2 + '3'
= (1 + 2) + '3'    ← 왼쪽부터 먼저
= 3 + '3'          ← 1+2는 둘 다 숫자 → 숫자 덧셈 → 3
= "33"             ← 3과 '3' → 문자열 모드 → "33"
```

문자열의 **위치**가 결과를 완전히 바꿔요. 비교해 보세요:

```js
1 + 2 + '3'    // "33"   — 숫자 덧셈 먼저, 그 다음 문자열 연결
'1' + 2 + 3    // "123"  — 처음부터 문자열 → 이후 전부 문자열 연결
1 + '2' + 3    // "123"  — 중간에서 문자열 만남 → 거기서부터 문자열
```

문자열을 만나는 **그 순간부터** 이후는 전부 문자열 연결이 돼요. 만나기 전까지는 숫자 덧셈이고요. 그래서 문자열이 어디에 있느냐가 중요해요.

## Q5. `NaN === NaN`의 결과는?

**보기**: `true` / `false`
**정답**: `false`

### 설명

JavaScript에서 **자기 자신과 같지 않은 유일한 값**이 바로 `NaN`이에요.

이건 JavaScript만의 결정이 아니에요. **IEEE 754 부동소수점 표준**에서 그렇게 정한 거예요. 이 표준은 C, Java, Python 등 거의 모든 프로그래밍 언어가 따르고 있어요.

왜 이렇게 정했을까요? NaN은 "Not a Number" — 계산할 수 없는 결과를 나타내는 값이에요. 문제는 NaN이 만들어지는 이유가 전부 다르다는 거예요:

```js
0 / 0            // NaN — 수학적으로 정의 불가
Math.sqrt(-1)    // NaN — 실수 범위에서 불가능
parseInt('abc')  // NaN — 숫자 파싱 실패
```

만약 `NaN === NaN`이 `true`라면, `0/0 === Math.sqrt(-1)`도 `true`가 돼요. "0을 0으로 나눈 것"과 "음수의 제곱근"이 같다? 수학적으로 말이 안 되죠. 그래서 IEEE 754 위원회(대표적으로 William Kahan)는 **NaN은 어떤 값과도 같지 않다**고 정한 거예요. 자기 자신을 포함해서요.

그러면 NaN인지 어떻게 확인할까요?

```js
// ❌ 오래된 방법 — 타입 변환 함정이 있음
isNaN('abc')          // true (!) 문자열인데도 true
isNaN(undefined)      // true (!) 이것도 true

// ✅ ES6 — 정확한 판별
Number.isNaN(NaN)     // true
Number.isNaN('abc')   // false — 타입 변환 없이 정확하게 판별

// ✅ Object.is — NaN을 같다고 판정하는 유일한 비교
Object.is(NaN, NaN)   // true

// 🧠 트릭 — NaN만 자기 자신과 같지 않다는 성질 이용
function myIsNaN(v) { return v !== v; }
```

## Q6. `typeof null`의 결과는?

**보기**: `"null"` / `"object"` / `"undefined"` / `"number"`
**정답**: `"object"`

### 설명

JavaScript에서 가장 유명한 버그예요. `null`은 명백히 객체가 아닌데 `typeof null`이 `"object"`를 반환해요.

이건 **1995년 초기 JavaScript 엔진의 C 구현에서 생긴 버그**예요.

당시 모든 값은 32비트로 저장됐어요. 하위 1~3비트를 **타입 태그**로 사용했는데:

| 타입 태그 | 값 | 의미 |
|----------|-----|------|
| `000` | 0 | object |
| `1` | 1 | int |
| `010` | 2 | double |
| `100` | 4 | string |
| `110` | 6 | boolean |

그리고 `null`은 C의 **NULL 포인터, 즉 전체 32비트가 0(`0x00`)**으로 표현했어요. 하위 비트도 당연히 `000`이죠. `typeof` 구현 코드에서 `null`을 별도로 먼저 체크하지 않았고, 타입 태그 `000`을 보고 "아, object구나"라고 판단해 버린 거예요.

ES6(2015년)에서 `typeof null === "null"`로 수정하자는 제안이 실제로 있었어요. 하지만 **이미 수많은 웹사이트가 `typeof null === "object"`에 의존**하고 있었기 때문에 기각됐어요. 웹의 하위 호환성 원칙 "Don't break the web"이 20년 된 버그를 영구적으로 고정시킨 셈이에요.

```js
// 그래서 null 체크에 typeof를 쓰면 안 된다
typeof null === 'object'  // true — 쓸모없음

// 정확한 null 체크
value === null             // 이게 맞음
```

## Q7. `'cat' && 'dog'`의 결과는?

**보기**: `true` / `false` / `"cat"` / `"dog"`
**정답**: `"dog"`

### 설명

`&&`은 불리언 연산자니까 당연히 `true`나 `false`를 반환할 거라고 생각하기 쉬워요. 하지만 JavaScript의 논리 연산자는 **불리언이 아닌 피연산자의 원래 값**을 반환해요.

`&&`의 규칙은 이거예요:
- 왼쪽이 **falsy** → 왼쪽 값 그대로 반환 (오른쪽은 보지도 않음)
- 왼쪽이 **truthy** → 오른쪽 값 반환 (오른쪽까지 봐야 결과를 알 수 있으니까)

`'cat' && 'dog'`에서 `'cat'`은 비어있지 않은 문자열이라 truthy예요. 그러면 오른쪽까지 평가해야 하니까 `'dog'`이 반환되는 거예요.

왜 이렇게 설계했을까요? 만약 `&&`가 항상 `true`/`false`만 반환했다면, **정보 손실**이 생겨요. 원래 값을 그대로 반환하면 불리언 판단도 되면서 동시에 **값으로도 활용**할 수 있거든요:

```js
// 조건부 실행 — &&가 원래 값을 반환하니까 가능한 패턴
isLoggedIn && showDashboard();

// 옵셔널 접근 — ?. 나오기 전에 쓰던 패턴
const city = user && user.address && user.address.city;

// ||도 마찬가지 — 기본값 패턴
const name = userInput || '익명';
```

불리언으로 변환하는 건 `if`문 같은 **호출한 쪽의 책임**이지, 연산자의 책임이 아닌 거예요.

```js
// 더 많은 예시
'cat' && 'dog'   // "dog"  — 'cat' truthy → 오른쪽
0 && 'dog'       // 0      — 0 falsy → 왼쪽에서 끝
'' && 'dog'      // ""     — '' falsy → 왼쪽에서 끝
'cat' && 0       // 0      — 'cat' truthy → 오른쪽(0)
```

## Q8. `0 ?? 'default'`의 결과는?

**보기**: `0` / `"default"` / `false` / `NaN`
**정답**: `0`

### 설명

이 문제는 `??`(null 병합 연산자)와 `||`(논리 OR)의 차이를 아는지 묻는 문제예요.

`||`였다면 결과가 달라져요:

```js
0 ?? 'default'   // 0         — ?? 사용
0 || 'default'   // "default" — || 사용
```

왜 다를까요?

- `||`는 왼쪽이 **falsy**이면 오른쪽을 반환해요. `0`은 falsy니까 `'default'`가 반환돼요.
- `??`는 왼쪽이 **null 또는 undefined**일 때만 오른쪽을 반환해요. `0`은 null도 undefined도 아니니까 `0` 그대로예요.

`??`가 ES2020에서 추가된 이유가 바로 이거예요. `||`로 기본값을 설정하면 `0`, `''`, `false` 같은 **유효하지만 falsy인 값**도 걸러버리는 문제가 있었거든요:

```js
// 사용자가 볼륨을 0으로 설정했는데...
const volume = userVolume || 50;  // 😱 0이 falsy라서 50으로 덮어씀!

// ??를 쓰면 null/undefined만 걸러냄
const volume = userVolume ?? 50;  // ✅ 0은 그대로 유지, null일 때만 50
```

정리하면:

| 연산자 | 오른쪽으로 넘어가는 조건 | falsy 값(0, '', false) 처리 |
|--------|------------------------|---------------------------|
| `\|\|` | 왼쪽이 **falsy** | 걸러버림 (오른쪽 반환) |
| `??` | 왼쪽이 **null/undefined** | 통과시킴 (왼쪽 반환) |

## Q9. `const x = (1, 2, 3); x`의 결과는?

**단답형**
**정답**: `3`

### 설명

괄호 안의 `,`는 **쉼표 연산자**예요. 배열의 쉼표나 함수 인자 구분 쉼표와는 다른, 진짜 연산자예요.

이 연산자는 JavaScript가 만든 게 아니에요. **C언어(1972년)에서 그대로 물려받은 것**이에요. Brendan Eich가 1995년에 JavaScript를 설계할 때 C/Java 문법을 의도적으로 차용했거든요. `for`, `while`, `if`, `++`, `--`, `? :` 같은 것들이 전부 C에서 온 건데, 쉼표 연산자도 그중 하나예요.

```c
// C언어 — 쉼표 연산자 (1972년부터 존재)
int x = (1, 2, 3);  // x == 3
for (i = 0, j = 10; i < j; i++, j--) { ... }
```

동작도 완전히 동일해요: **왼쪽부터 순서대로 평가하고, 마지막 값을 반환**해요.

```js
// JavaScript — C와 동일한 동작
const x = (1, 2, 3);  // 1 평가 → 2 평가 → 3 반환
console.log(x);       // 3
```

### 왜 마지막 값을 반환할까?

"왜 첫 번째도 아니고 전부도 아니고 마지막이야?" 라고 물을 수 있는데요.

순차 실행에서 앞의 것들은 **부수 효과(side effect)를 위한 것**이고, "결과"는 마지막 표현식이 자연스럽기 때문이에요. 이건 쉼표 연산자만의 개념이 아니라, 많은 언어에서 **"블록의 값 = 마지막 표현식"**이라는 동일한 설계를 따르고 있어요:

```js
// JavaScript의 eval도 마지막 값을 반환
eval('1; 2; 3')  // 3
```

```ruby
# Ruby — 블록의 마지막 표현식이 반환값
x = begin; 1; 2; 3; end  # x == 3
```

첫 번째를 반환한다면 나머지를 왜 실행한 건지 의미가 없고, 전부 반환하면 배열이 되어야 해서 연산자라고 부르기 어렵죠. 마지막 값이 유일하게 자연스러운 선택이에요.

### 그런데 왜 C에서 이걸 만들었을까?

C에서 쉼표 연산자가 필요했던 가장 큰 이유는 **`for`문** 때문이에요. `for`문의 구조를 보면:

```
for (초기화식; 조건식; 갱신식) { ... }
```

각 슬롯은 **표현식(expression) 하나**만 받도록 설계돼 있어요. 왜 문(statement)을 허용하지 않았을까요? 문을 허용하면 **세미콜론이 뭘 구분하는 건지 모호해지기 때문**이에요:

```js
// 만약 문(statement)을 허용했다면?
for (let i = 0; i < 5; i++; j--)
//                          ^ 이 세미콜론이 슬롯 구분인가? 문의 끝인가?
//                            파서가 구분할 수 없음
```

표현식만 허용하면 `;`이 명확하게 슬롯 구분자 역할을 해요. 그래서 갱신 슬롯에서 여러 동작을 하려면 **문을 나열할 수 없고, 표현식 하나로 합쳐야** 하니까 쉼표 연산자가 필요했던 거예요:

```js
// ❌ 이렇게 못 함
for (let i = 0; i < 5; i++; j--)

// ✅ 쉼표 연산자로 하나의 표현식으로 합침
for (let i = 0, j = 10; i < j; i++, j--) {
  console.log(i, j);
}
// 출력: 0 10, 1 9, 2 8, 3 7, 4 6
```

정리하면: **C에서 `for`문의 파서 모호성을 해결하기 위해 만들어진 연산자 → JavaScript가 C 문법을 차용하면서 그대로 가져온 것**이에요.

`for`문 외에는 코드 압축기(minifier)가 여러 문장을 한 줄로 합칠 때 정도 쓰여요. 실무 코드에서 직접 쓸 일은 거의 없어요.

### 같은 `,`인데 역할이 다르다

마지막으로 주의할 점 — 같은 `,`여도 위치에 따라 역할이 완전히 달라요:

```js
let a = 1, b = 2;     // 변수 선언의 쉼표 (연산자 아님)
foo(1, 2);             // 함수 인자 구분 (연산자 아님)
[1, 2, 3];             // 배열 요소 구분 (연산자 아님)
const x = (1, 2, 3);   // 쉼표 연산자 ✓
```

## Q10. `~~3.7`의 결과는?

**보기**: `3` / `4` / `3.7` / `NaN`
**정답**: `3`

### 설명

JavaScript에서 비트 연산자를 쓸 때 재미있는 일이 벌어져요. JavaScript의 모든 숫자는 **64비트 부동소수점(double)**인데, 비트 연산을 할 때만 **일시적으로 32비트 정수(Int32)로 변환**해요. 연산이 끝나면 다시 부동소수점으로 돌아오고요.

`~`는 비트 NOT 연산자예요. 모든 비트를 뒤집어요. 한 번 뒤집고(`~`) 다시 뒤집으면(`~~`) 원래 비트로 돌아오지만, **그 과정에서 32비트 정수로 변환될 때 소수점이 잘려나가요**.

```
3.7
→ 32비트 정수 변환: 3 (소수점 버림)
→ ~3 = -4 (비트 NOT)
→ ~(-4) = 3 (비트 NOT 한 번 더)
→ 결과: 3
```

이게 실무에서 `Math.trunc()` 대신 쓰이는 트릭이에요:

```js
~~3.7      // 3
~~(-3.7)   // -3
3.7 | 0    // 3   — 이것도 같은 원리 (OR 0은 값을 안 바꾸지만 Int32 변환은 일어남)
Math.trunc(3.7)  // 3 — 정석적인 방법
```

하지만 **32비트 정수 범위(약 ±21억)를 넘으면 깨져요**:

```js
~~(2 ** 31 - 1)  // 2147483647 — OK (32비트 최대값)
~~(2 ** 31)      // -2147483648 — 😱 오버플로!
~~(2 ** 53)      // 0 — 완전히 망가짐

Math.trunc(2 ** 53)  // 9007199254740992 — 이건 정확
```

그래서 `~~`나 `| 0` 트릭은 성능이 중요하고 값이 작다고 보장되는 경우에만 쓰고, 일반적으로는 `Math.trunc()`가 안전해요.
