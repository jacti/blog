실험 1 — 불변성은 엔진이 실제로 차단하는가
node examples/ch11/03-immutability-proof.js
node examples/ch11/03-immutability-proof.js # "use strict" 모드에서 문자열 수정 시도 str[0] = 'H' → TypeError: Cannot assign to read only property '0' of string 'hello' arr[0] = 'H' → Hello (성공)
문자열의 각 인덱스는 read-only property다. strict mode에서는 TypeError, 아니면 조용히 무시된다.
배열은 객체이므로 자유롭게 수정된다. 이것이 "원시 값은 불변"의 실체.
실험 2 — 같은 "hello"인데 내부가 다르다
node --allow-natives-syntax examples/ch11/02-interning-compact.js
node --allow-natives-syntax examples/ch11/02-interning-compact.js === 문자열 인터닝 실험 === 리터럴 'hello' : true 같은 리터럴 (변수2) : true 'hel' + 'lo' : false 'hello world'.slice : false JSON.parse : true === 그런데 === 비교는? === literal1 === dynamic : true literal1 === sliced : true literal1 === parsed : true
%IsInternalizedString()은 V8 내부 함수로, 일반 JS에서는 호출할 수 없다.
리터럴은 String Table에 등록(인터닝)되지만, 동적 생성 문자열은 안 된다.
그럼에도 === 비교는 항상 정확하다 — 인터닝은 성능 최적화이지, 정확성에 영향 없음.
