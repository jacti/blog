# 1차 개발 계획 (업데이트)

## 현재 파악한 프로젝트 상태
- Hextra 스타터 템플릿을 Git Submodule 방식으로 포함하고 있으며 `themes/hextra/`에 테마 자산이 정상 존재함.
- 빌드 시 원격 리소스(FlexSearch) 로드가 네트워크 제한에 막혀 실패할 수 있으므로 오프라인 대안을 검토해야 함.
- 현재 콘텐츠는 홈/Docs/About 데모 수준으로 실제 섹션 구조가 아직 반영되지 않음.
- 다국어(i18n) 설정은 준비만 되어 있고 기본 언어는 한국어로 확장 예정.

## 합의된 정보 구조 및 섹션 목적
- `Learning log`(`/learning-log/`): AI와 함께 정리한 학습 노트를 문서형으로 제공.
- `Debug notes`(`/debug-notes/`): 트러블슈팅 사례와 해결 과정을 스토리텔링.
- `Playground`(`/playground/`): AI 제작 데모/프로토타입을 카드형 목록으로 공개하고 개별 HTML 페이지로 연결.
- `About`(`/about/`): 자기소개, 경력, 연락처를 통합한 프로필 페이지.
- `Workbench`: `content/workbench/`에 초안과 실험 콘텐츠를 `draft: true`로 유지.

## 레이아웃 및 UI 방향
- `Learning log`, `Debug notes`: Hextra docs 레이아웃(`type: docs`)을 유지해 좌측 사이드바와 우측 TOC를 제공.
- `Playground`: 리스트 페이지는 카드 그리드, 각 데모는 HTML 임베드 또는 정적 페이지로 구성.
- `About`: 히어로 소개 + 경력/스킬 요약 + 연락 섹션으로 자기PR 중심 구성.
- 전반적으로 높은 가독성을 우선하며 필요 시 포인트 컬러만 조정.

## 색상 관련 메모
- 현재 배포본은 흰색/짙은 남청 위주의 모노톤으로, 가독성은 우수함.
- Tailwind 기반 CSS 변수(`assets/css/custom.css`)로 라이트/다크 모드 팔레트를 조정 예정.
- 후보 컬러 조합: 슬레이트+틸, 웜그레이+앰버, 인디고+시아. 최종 선택은 후속 논의.

## 향후 작업 일정 (우선순위 순)
1. **현재 테마 정상 동작 확인**: `hugo server --buildDrafts --disableFastRender`로 로컬 동작 점검. (네트워크 제한으로 FlexSearch 로딩 실패 가능 → 추후 오프라인 번들 고려)
2. **다국어 설정 추가**: `defaultContentLanguage: ko`, `languages` 블록 정의, `i18n/ko.yaml` & `i18n/en.yaml`에 메뉴/레이블 번역 작성.
3. **콘텐츠 구조 개편**: `content/learning-log/`, `content/debug-notes/`, `content/playground/`, `content/workbench/` 생성 및 각 `_index.md` 초안 작성.
4. **Playground 레이아웃 설계**: 내장 `cards` 숏코드 등 테마 컴포넌트를 우선 활용해 카드 그리드 구현, 필요 시 파셜 오버라이드 고려.
5. **About 페이지 확장**: 자기소개, 경험 하이라이트, 연락 섹션을 포함하는 콘텐츠 및 레이아웃 구성.
6. **색상 팔레트 커스터마이징**: `assets/css/custom.css`에서 CSS 변수 기반으로 라이트/다크 모드 색상 정의.
7. **샘플 콘텐츠 및 컴포넌트 구현**: 각 섹션에 대표 예시 작성 후 피드백 반영.
8. **배포 준비**: `hugo --buildDrafts --buildFuture --minify`로 사전 검증, FlexSearch 등 원격 의존성 처리 후 자동 배포 파이프라인 점검.

## 주의 사항
- Git Submodule과 Hugo Modules 방식은 혼용 금지. `hugo.yaml`의 `theme: hextra` 설정을 유지하고 `module.imports`는 사용하지 않음.
- 네트워크 제한 환경에서는 원격 JS/CSS 가져오기 실패 가능 → 필요한 리소스를 `static/`에 사전 배포하는 방안 검토.
- `assets` 커스터마이징은 CSS 변수 위주로 진행하고 SCSS 파일은 생성하지 않음.
- `Workbench` 이하 콘텐츠에 `draft: true`를 유지해 프로덕션 빌드에 포함되지 않도록 관리.
